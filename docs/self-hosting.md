# Self-Hosting Guide — llmeval

This guide walks through deploying the complete `llmeval` stack to production cloud hosting platforms or running it locally on your own hardware.

---

## 1. Requirements

Before deploying, ensure you have:
- **Docker & Docker Compose**: Installed locally for building containers and running local stacks.
- **Database**: Either a free [Supabase](https://supabase.com) project or a standard PostgreSQL 16+ database instance.
- **LLM Judge API**: An OpenAI API key (or an OpenAI-compatible endpoint such as LiteLLM, Azure OpenAI, or self-hosted Ollama/vLLM) for RAGAS evaluation scoring.

---

## 2. Supabase Setup

If you prefer managed PostgreSQL with an administrative GUI:

1. Create a free organization and project at [supabase.com](https://supabase.com).
2. Once your database is provisioned, navigate to the **SQL Editor** from the left sidebar (`[Screenshot: Supabase sidebar highlighting SQL Editor icon]`).
3. Click **New query**, paste the entire contents of `supabase/migrations/001_initial_schema.sql`, and click **Run** (`[Screenshot: SQL Editor running migration script successfully with green confirmation toast]`).
4. Navigate to **Project Settings -> API** (`[Screenshot: Project Settings API keys page]`).
5. Copy two values:
   - **Project URL** (e.g., `https://abcdefghijklm.supabase.co`) -> Set as `SUPABASE_URL`.
   - **service_role secret** (hidden by default, click Reveal) -> Set as `SUPABASE_SERVICE_KEY`.

> [!CAUTION]
> Never expose your `service_role` key to public clients or commit it to source control. It bypasses Row Level Security.

---

## 3. Environment Variables

Below is the exhaustive list of environment variables used across the backend, frontend, and runner.

| Variable Name | Service | Secret? | Safe to Commit? | Description |
| :--- | :--- | :--- | :--- | :--- |
| `SUPABASE_URL` | Backend | No | No | Supabase project API endpoint URL |
| `SUPABASE_SERVICE_KEY` | Backend | **Yes** | No | Supabase service role admin key |
| `DATABASE_URL` | Backend | **Yes** | No | Alternative to Supabase creds: standard PostgreSQL connection string (`postgresql://user:pass@host:5432/db`) |
| `LLMEVAL_API_KEY` | Backend / Runner | **Yes** | No | Shared secret key for authenticating API requests |
| `NEXT_PUBLIC_API_URL` | Frontend | No | Yes | Publicly accessible URL of the backend API (e.g., `http://localhost:8000` or `https://api.yourdomain.com`) |
| `NEXT_PUBLIC_API_KEY` | Frontend | No* | No | Passed to browser bundle to authenticate frontend API requests (*see note in README) |
| `OPENAI_API_KEY` | Runner | **Yes** | No | API key used by RAGAS judge model |
| `PIPELINE_API_KEY` | Runner | **Yes** | No | Optional API key sent in header to your LLM pipeline endpoint |

## 4. Deploying Backend to Railway

Railway is ideal for running the containerized FastAPI service:

1. Push your repository to GitHub.
2. Log in to [Railway.app](https://railway.app) and click **New Project -> Deploy from GitHub repo**.
3. Select your repository and specify the Root Directory as `backend`.
4. Under **Variables**, add:
   - `SUPABASE_URL`: Your Supabase project URL (or `DATABASE_URL` if using Railway Postgres).
   - `SUPABASE_SERVICE_KEY`: Your Supabase service role key.
   - `LLMEVAL_API_KEY`: Generate a secure random string (e.g. via `openssl rand -hex 32`).
5. Under **Settings -> Networking**, click **Generate Domain** to get your public API URL (e.g., `https://llmeval-api.up.railway.app`).

---

## 5. Deploying Frontend to Vercel

1. Log in to [Vercel.com](https://vercel.com) and click **Add New -> Project**.
2. Import your GitHub repository.
3. Set the **Root Directory** to `frontend`.
4. Under **Environment Variables**, configure:
   - `NEXT_PUBLIC_API_URL`: The domain generated by Railway in Section 4 (e.g., `https://llmeval-api.up.railway.app`).
   - `NEXT_PUBLIC_API_KEY`: The exact same `LLMEVAL_API_KEY` string you configured in Railway.
5. Click **Deploy**. Your dashboard will be live on a `.vercel.app` URL.

## 6. Running the Runner Locally

To manually trigger runs from your development workstation against your deployed cloud backend:

```bash
# 1. Install CLI
cd runner
pip install -e .

# 2. Export environment variables
export LLMEVAL_API_KEY="your-secret-api-key"
export OPENAI_API_KEY="sk-..."

# 3. Execute evaluation loop
llmeval run --config path/to/your/.evalrc.yml
```

## 7. Running the Full Stack Locally Without Cloud Services

If you do not want to use Supabase or external cloud providers, you can spin up a fully isolated local stack using PostgreSQL inside Docker:

```bash
# Run the local-only compose stack
docker-compose -f docker-compose.local.yml up --build -d
```

### What happens under the hood:
- Spawns a `postgres:16-alpine` container (`db`) on port `5432`.
- Automatically initializes the database schema by mounting `001_initial_schema.sql` into `/docker-entrypoint-initdb.d/`.
- Configures the backend service to use `DATABASE_URL=postgresql://llmeval:llmeval@db:5432/llmeval` instead of Supabase credentials.
- Backend uses the native `LocalPostgresAdapter` built into `api.db` to execute queries directly via psycopg2.

Access the services at:
- **Dashboard**: http://localhost:3000
- **API**: http://localhost:8000

## 8. Troubleshooting

### Common Errors and Solutions

#### 1. `HTTP 401 Unauthorized` when running `llmeval run` or opening Dashboard
- **Cause**: The `LLMEVAL_API_KEY` in your `.env` or runner session does not match the key configured on the backend.
- **Solution**: Check that `api_key` in `.evalrc.yml` or `LLMEVAL_API_KEY` environment variable matches the backend container's `LLMEVAL_API_KEY`.

#### 2. `ValueError: Either DATABASE_URL or (SUPABASE_URL and SUPABASE_SERVICE_KEY) must be provided.`
- **Cause**: The backend container started without database connection credentials.
- **Solution**: Ensure your `.env` file exists and contains valid Supabase variables, or set `DATABASE_URL` if running against local PostgreSQL.

#### 3. `Failed to connect to pipeline endpoint` / `Timeout` during evaluation
- **Cause**: The runner could not reach the pipeline URL specified in `.evalrc.yml`.
- **Solution**: If running `llmeval run` inside Docker against a local server running on your host machine, use `http://host.docker.internal:PORT` instead of `localhost`. Increase `pipeline.timeout_seconds` if your LLM generation takes longer than 30s.

#### 4. `EINVAL: invalid argument, readlink ... .next` on Windows
- **Cause**: Next.js build cache or symlinks conflicted on Windows synced folders (OneDrive).
- **Solution**: Run `rmdir /s /q frontend\.next` and run `npm run build` again.
